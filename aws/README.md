# AWS Backup Infrastructure for Atlanta Gleaner

This directory contains AWS Lambda and EventBridge configuration for automated news feed refresh as a backup to the primary Vercel cron job.

## Overview

### Why AWS Lambda Backup?

The primary news feed refresh runs on Vercel cron daily at **5 AM UTC**. To ensure availability even if Vercel fails, a secondary Lambda function runs at **5 PM UTC** (12 hours later) and writes to the same Vercel Blob storage.

**Fallback chain** (in `/api/news` route):
1. **live** cache (fresh from Vercel cron)
2. **previous** cache (archived before last publish)
3. **backup** cache (from AWS Lambda)

If Vercel cron fails, the API will serve previous or backup cache instead of an empty feed.

### Architecture

```
Vercel Cron (5 AM UTC)           AWS Lambda (5 PM UTC)
         |                               |
         ├─→ build feed                 ├─→ build feed
         ├─→ save to Blob               ├─→ save to Blob
         └─→ update Edge Config         └─→ update Edge Config
                  |                             |
                  └─────────→ Vercel Blob ←────┘
                       (shared storage)
                             |
                             ↓
                        /api/news route
                    (fallback chain: live → previous → backup)
```

## Lambda Function

### Source: `lambda/news-feed-refresh.ts`

A standalone TypeScript/Node.js function that:

1. **Builds News Feed**:
   - Queries Serper API for 6 editorial topics (Atlanta, Georgia, CNN, business, tech, real estate)
   - Fetches Spotify episodes (Audio Dispatch)
   - Deduplicates and scores items
   - Returns top 20 items

2. **Saves to Vercel Blob**:
   - Uploads cache entry as `news-feed/backup.json`
   - Returns public Blob URL

3. **Updates Edge Config**:
   - Writes reference to `news_cache_backup` key in Edge Config
   - Allows `/api/news` route to access the backup

4. **Logging & Errors**:
   - All operations logged to CloudWatch
   - Comprehensive error messages for troubleshooting
   - Function returns JSON response (success/failure)

### Environment Variables

| Variable | Required | Source |
|----------|----------|--------|
| `SERPER_API_KEY` | Yes | Vercel env |
| `SPOTIFY_CLIENT_ID` | Yes | Vercel env |
| `SPOTIFY_CLIENT_SECRET` | Yes | Vercel env |
| `SPOTIFY_SHOW_IDS` | Yes | Vercel env (comma-separated, up to 8) |
| `BLOB_READ_WRITE_TOKEN` | Yes | Vercel Blob API |
| `EDGE_CONFIG_ID` | Yes | Vercel project settings |
| `VERCEL_API_TOKEN` | Yes | Vercel account settings |
| `YOUTUBE_API_KEY` | No | Vercel env (optional, fallback to scrape) |

### Execution Details

- **Runtime**: Node.js 18.x (or later)
- **Memory**: 512 MB
- **Timeout**: 60 seconds
- **Typical duration**: 2-5 seconds (most time spent on API calls)
- **Cost**: Free tier (1 invocation/day = 30/month)

## EventBridge Schedule

The Lambda function is triggered by EventBridge **daily at 5 PM UTC**:

```
Cron expression: 0 17 * * ? *
```

To adjust schedule:
- **5 PM UTC** (default): `0 17 * * ? *`
- **12 PM UTC** (noon): `0 12 * * ? *`
- **6 PM UTC**: `0 18 * * ? *`

Timezone always **UTC** in EventBridge (convert your local time).

To test that `/api/news` uses backup cache:

## Troubleshooting

### Common Issues

| Issue | Cause | Fix |
|-------|-------|-----|
| **Function Timeout** | Slow APIs (Serper, Spotify) | Increase timeout to 120s, or reduce queries |
| **"Missing API Key"** | Env vars not set | Verify all vars in Lambda config |
| **Blob Upload Fails (403)** | Invalid token | Get fresh `BLOB_READ_WRITE_TOKEN` from Vercel |
| **EventBridge Not Triggering** | Rule disabled or wrong time | Check rule state, verify cron expression timezone |
| **Feed Not Using Backup** | Vercel cron still working | Check `/api/news` response for `source` field |

See `SETUP.md` "Troubleshooting" section for detailed fixes.

## Rollback / Disable

If Lambda causes issues, quickly disable scheduled trigger:

```bash
aws events put-rule \
  --name atlanta-gleaner-news-refresh-schedule \
  --state DISABLED
```

Feed continues using **Vercel cron only** (primary), fallback chain still works.
